> 作成日: 2026-06-02 / 対象: LINE Harness CRM（line.oryzae.shop）
> 目的: 誕生日当日に「誕生日クーポン」を自動配信し、ポイント活用促進＋EC売上＋ファン化を狙う
> 運用主体: 小泉さんが判断（割引率等）、Claude が実装・運用

## 1. 確定仕様（2026-06-08 最終確定：直近注文額で出し分け）

| 項目 | 内容 |
|---|---|
| **トリガー** | 誕生日**当日**（毎日 0:00 JST の日次cronで「今日が誕生日」の人を抽出）|
| **判定軸** | **直近の注文額**が 5,000円未満 / 以上（厳密AOVは注文回数データが無く重いため代用。「今の買い方」を反映）|
| **特典（出し分け）** | 直近5,000円未満 → **送料無料クーポン（条件なし）** / 5,000円以上 → **500円OFFクーポン（条件なし）** |
| **有効期限** | 誕生日当日から **14日間**（`starts_at`=誕生日, `ends_at`=+14日）|
| **配信チャネル** | LINEのみ（メール代替配信はしない） |
| **対象** | LINE連携後に誕生日登録した顧客（`loyalty_points.birthday` に保存済み、かつフォロー中） |
| **付与回数** | 年1回（同一年に二重発行しない冪等制御）|
| **文面方針** | 500円OFFは素直に「500円OFF」とだけ伝える。**送料無料ラインには触れない**（「5,000円以上で送料無料」と書くと5,300円カートを誘発し逆効果）|

### なぜ「直近注文額で出し分け」なのか（中核ロジック）
- **送料無料クーポンは「1回5,000円未満で買う人」にしか価値がない**（5,000円以上は元々送料無料＝無価値）
- 「全員に送料無料」だと、まとめ買いの優良顧客に無価値な特典を送り「私には使えない」と逆効果
- 判定は本来 AOV（1回あたり購入額）が理想だが、CRMに注文回数データが無く厳密計算は重い。→ **直近の注文額**で代用（「今の買い方」を反映でき実装が軽い）
- ランク（=累計購入額）は「毎回少額のヘビーリピーター」を高ランク扱いし送料無料を渡せなくなるのでズレる→不採用

### ⚠️ 500円OFF側の送料復活リスク（許容と決定）
- 5,000円以上層が「条件なし500円OFF」を使い、割引後5,000円を切ると送料480円が復活する
- 事故るのは「5,000〜5,500円で買う狭い帯」のみ（6,000円以上 or 送料無料側は無関係）
- 「誕生日は条件なしで気持ちよく」を優先し**利用条件は付けない**＝送料復活は**許容**する判断
- **文面では送料無料ラインに触れない**。「5,000円以上で送料無料」と書くと5,300円カート（事故帯ど真ん中）を誘発し逆効果になるため、500円OFFは素直に「500円OFF」とだけ伝える

### 各特典の根拠
- **送料無料クーポン**: 既存 `welcome`（`DiscountCodeFreeShipping`・最低条件なし・1顧客1回・累計5,949回稼働）と同型。5,000円以上自動送料無料とは**コンフリクトしない**（Shopifyは最安レート自動採用、二重マイナス/エラーなし）
- **500円OFFクーポン**: 既存 `redeem` の price_rule発行ロジック（fixed_amount・顧客限定・1回限り）を流用。`ends_at`を+14日に設定

## 2. 実装方針（既存資産の流用）

### 流用できるもの（ゼロから作らない）
- **クーポン発行ロジック**: `loyalty.ts` の `redeem`（POST `/api/loyalty/shopify/:id/redeem`）が、Shopify `price_rules` + `discount_codes` API で顧客限定・1回限りの割引コードを動的発行している。これを流用。
- **誕生日データ**: 誕生日登録時に Shopify `facts.birth_date` と自社D1 `loyalty_points.birthday` へ保存済み
- **日次cron基盤**: `scheduled()` の `0 0 * * *` ブランチに1ジョブ追加するだけ
- **LINE配信**: 既存の誕生日登録通知（loyalty.ts 957-1112行付近）のLINE push経路を流用
- **配信時間枠制御**: 9-23時JST（システム自動）

### 新規に作るもの（🟢 新規ファイル、🔴保護ゾーンの既存コードは書き換えない）
1. **`src/services/birthday-cron.ts`**（新規）
   - 今日の月日（MM-DD）に一致する `loyalty_points.birthday` を持ち、LINE連携済み・フォロー中の顧客を抽出
   - 各人にクーポン発行（既存redeemロジックをベースに `ends_at` と percentage 対応を追加）
   - LINE Flexで配信。LINEで送れない場合もメールは送らない
   - 冪等性: 「この顧客にこの年の誕生日クーポンを発行済みか」を記録するテーブル/メタフィールドでガード
2. **`scheduled()` への1行追加**（`0 0 * * *` ブランチに `jobs.push(processBirthdayCoupons(env))`）
   - ⚠️ index.ts は🟡確認必須ゾーン。この1行追加は事前に小泉さんへ確認してから。

### 既存redeemに足りない要素（誕生日クーポン用に追加）
- `ends_at`: 誕生日+14日（現状redeemは終了日未指定）
- `value_type: 'percentage'` 対応（%OFFにする場合。現状は fixed_amount）
- prerequisite_customer_ids でその顧客限定（既存同様）

## 3. 冪等性・事故防止（重要）

- **二重発行防止**: 同一顧客・同一年に1回のみ。発行履歴を記録し、cron再実行や複数回起動でも重複しないようにする。
- **登録ボーナス(100pt)との区別**: 登録時100ptは「個人情報提供のお礼」で別物。誕生日クーポンは「毎年のお祝い」。混同しない。
- **うるう年(2/29生まれ)**: 平年は2/28 or 3/1のどちらで配信するか決める（推奨: 2/28）。
- **配信失敗時**: メール代替配信はしない。失敗理由を `birthday_coupon_log.error_message` に残して運用確認する。
- **テスト**: 本番投入前に、自分（小泉さん）の誕生日を一時的にtoday設定して動作確認。
