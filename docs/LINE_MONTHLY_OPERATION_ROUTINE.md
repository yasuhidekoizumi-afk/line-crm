# LINE Harness 月次運用ルーティン

> 作成日: 2026-06-02 / 対象: line.oryzae.shop（LINE公式アカウント CRM）
> 運用主体: **小泉さんが判断、Claude が作業**
> 設計思想: 「毎月開いて何を判断するか」を起点に、「誰がいなくなっても回る」仕組みにする

---

## 0. なぜこのルーティンが必要か（診断結果サマリ）

2026-06-02 時点の実態：

| 指標 | 数値 | 評価 |
|---|---|---|
| 友だち総数 | **20,129人** | 資産は十分 |
| ロイヤルティ会員 | 13,105人（67,899pt付与 / 170,800pt償還） | 活用余地大 |
| 本番ブロードキャスト | **0件**（テスト2件のみ） | ❌ 配信が稼働していない |
| ステップ配信シナリオ | 2本あるが stepCount=0（空） | ❌ 中身が無い |
| タグ | 6個あるが「新規顧客」タグは0人 | ⚠️ 付与運用が未整備 |

**結論：2万人いるのに企画配信が一度も本番稼働していない。箱（タグ・シナリオ・ロイヤルティ）は揃っているので、回す仕組みを作れば即動く。**

## 1. 配信方針（このルーティンの軸）

- **頻度：週1回ペース**（月4〜5配信）。やりすぎてブロックされるのが最悪なので、継続できるペースを最優先。
- **目的の4本柱**（毎回どれか1つを主目的にする）：
  1. **EC売上**（新商品告知・セール・リピート促進）
  2. **ポイント活用促進**（失効前告知・償還促し）※有効期限365日
  3. **関係性・ファン化**（発酵/健康コンテンツ・ブランドストーリー）
  4. **新規友だちの育成**（ウェルカムステップ）
- **配信時間枠：9:00-23:00 JST**（システムが自動制御。深夜予約は翌朝9時に自動延期）
- **推奨配信枠：火 or 木の 11:00 / 19:00 前後**（昼休み・帰宅後）

### 4週ローテーション例（毎月使い回せる型）
| 週 | 主目的 | 配信内容の型 |
|---|---|---|
| 第1週 | EC売上 | 新商品 or 今月の主力商品の訴求 |
| 第2週 | ファン化 | 発酵・健康コラム/レシピ（売り込まない） |
| 第3週 | EC売上 | セール/限定/リピート促進 |
| 第4週 | ポイント活用 | 失効前告知・ポイント使ってお得に |

## 2. 月次ルーティン（小泉さんが毎月開いて判断すること）

### 📅 月初（毎月1日 or 第1営業日）── 所要 15分

**小泉さんの判断：「今月、何を4回送るか」を決める**

Claude が事前に用意するもの（小泉さんは選ぶだけ）：
1. **前月実績レポート**（後述の指標）
2. **今月の配信ネタ候補リスト**（新商品予定・在庫状況・季節イベントから3〜5案）
3. 各案のドラフト原稿（叩き台）

→ 小泉さんは「これとこれを送る」「この週は休む」を決める。

### 📅 配信前日 ── 所要 5分

**小泉さんの判断：「この原稿で送ってOKか」**

Claude が用意：
- 仕上げた配信原稿（テキスト or Flex）
- 配信対象セグメント（全員 / 特定タグ）
- 予約日時

→ 小泉さんが OK を出したら Claude が予約投入。

### 📅 配信翌日 ── 所要 3分（任意）

Claude が配信結果（到達・クリック・コンバージョン）を報告。次の配信に反映。

## 3. 毎月見る指標（レポートの中身）

毎月、Claude が MCP/API 経由で自動取得して報告：

| 指標 | 取得元 | 判断materりal |
|---|---|---|
| 友だち増減 | `/api/friends/count` | 純増しているか |
| ブロック率 | LINE API stats | 配信過多になっていないか（最重要の歯止め） |
| 配信到達/クリック | broadcast の successCount | 内容が刺さっているか |
| ポイント償還額 | `/api/loyalty/stats` | ロイヤルティが機能しているか |
| ランク分布 | `/api/loyalty/stats` byRank | VIP層が育っているか |
| Shopify売上連動 | Shopify MCP | LINE配信が売上に効いたか |

**判断の歯止め：ブロック率が前月比で急増したら頻度を下げる。** これだけは絶対守る（2万人の資産を守るため）。

## 4. 仕込みタスク（ルーティンを回す前の初期整備）

ルーティンを「誰がいなくても回る」状態にするため、最初に1回だけやる整備：

- [ ] **T1. タグ運用の整備** — 「新規顧客」「購入済み」「リピーター」タグが自動付与されるようにする（現状0人＝付与ロジックが未稼働の疑い）。Shopify Webhook 連携の確認。
- [ ] **T2. ウェルカムステップ配信の中身を入れる** — 空の「ウェルカムフロー」シナリオに2〜3ステップ（友だち追加→当日お礼+ブランド紹介→3日後初回クーポン）を投入。新規友だちが自動で育つ状態に。
- [ ] **T3. 配信原稿テンプレート集を作る** — 4本柱 × 各2〜3パターンのテンプレを `FERMENT_TEMPLATES.md` ベースで用意。毎回ゼロから作らない。
- [ ] **T4. 失効前ポイント償還配信の仕組み** — 有効期限365日に対し、失効30日前の保有者を抽出→償還促し配信する月次ジョブ（半自動）。
- [ ] **T5. MCP接続の恒久化** — `.mcp.json` のAPIキーをリテラル設定済み（git追跡解除済み）。MCP再起動で `account_summary` 等が使えることを確認。

## 6. 技術メモ（実行者向け）

- API ベースURL: `https://oryzae-line-crm.oryzae.workers.dev`
- 認証: `Authorization: Bearer <API_KEY>`（`.mcp.json` に設定済み、git追跡外）
- ⚠️ **Worker本体（`apps/worker/`）は🔴絶対保護ゾーン。配信運用はAPI/MCP経由のみで、コードは触らない。**
- 配信投入: `POST /api/broadcasts` → `POST /api/broadcasts/:id/send`（or `/send-segment`）
- 既存タグID: VIP / リピーター / 新規顧客 / 購入済み / 388pt_campaign / テスト送信
- ポイント設定: 還元率1% / 1pt=1円 / 有効期限365日 / 新規登録ボーナス300pt
