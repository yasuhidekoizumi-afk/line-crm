## 2. Harness「固有」の強み（公式に無い・重複しない）

これがHarnessを残す理由。ここが本体価値:

- **ロイヤルティポイント管理**（`/api/loyalty/stats`）— 還元1%・ランク制・13,105会員・171,200pt償還実績。**公式には無い。** Shopify連携の中核。
- **顧客名寄せ / CRM**（LINE×Shopify×メールを顧客IDで統合）
- **API/MCP自動化**（Claudeがプログラムで全操作できる。公式はGUI手動が基本）
- **きめ細かいセグメント**（metadata条件・ref_code等）

## 3. 重複が生む実害（「誰がいなくなっても回る」哲学に反する）

- タグを両方で別々に付与 → どちらが正か分からなくなる
- 配信を両方から打てる → 二重送信・通数二重消費のリスク
- 担当が「どっちで作業するか」迷う → 属人化・引き継ぎ不能

## 4. 役割分担の論点（次に決めること）

棚卸しから見える筋:
- **配信実行レイヤー**（実際にLINEに飛ばす）は、通数管理・上限・課金が公式に紐づく以上、**公式が物理的に握っている**。Harnessの`/api/broadcasts`も内部でLINE Messaging APIを叩く＝結局同じ通数枠を消費しているはず（要検証）。
- だとすると「Harnessに一本化」しても通数上限問題からは逃げられない。**プラン/通数の問題は公式アカウント単位の制約。**
- **Harnessの値は「配信を代替すること」ではなく「ポイント×CRM×自動化で、誰に何を送るかを賢くすること」。** 配信の実行は公式に寄せ、Harnessは頭脳に徹する、が素直。

## 4.5 【調査確定】友だち数乖離の真因 = 2026-04-09 一括インポートのデータバグ

**結論: 「ブロック多発で離脱」ではない。データ移行バグ。**

調査でわかったこと（2026-06-03、Harness API直叩き）:
- Harness DBの友だちは**圧倒的多数が `2026-04-09` 登録** = 旧データの一括インポート跡。以降は1日数件の自然流入。
- `isFollowing=false` が **offset約1,000〜13,000の「連続した塊」** で100%。自然なブロックなら時期/属性でバラけるはずで、こんな連続にはならない。
- offset 0付近（5〜6月の自然流入）と offset 13,500以降は following 94〜100% で健全。
- → **2026-04-09の一括インポートで、約1万件の `isFollowing` が誤って false で取り込まれた**と判断。

**実害:**
- `count` API は isFollowing フィルタを無視して全件(20,134)を返す → 数字が水増しに見える。
- 真の配信対象 ≒ 公式の **15,126人** が正。
- **Harnessでセグメント/シナリオ配信を組むと、誤ってfalseになった約1万人が配信対象から漏れる。** ウェルカムフロー以前に土台データが壊れている。

**対策候補（未実施・要判断）:**
1. LINE Messaging APIの最新フォロー状態でHarness DBの isFollowing を再同期するバッチ（Worker側=🔴保護ゾーン、要承認）
2. or 公式を配信の正とし、Harnessは「届ける」役を持たず「誰に送るか決める頭脳」に徹する（役割分担で回避）

## 4.7 【判断材料】「将来Harnessに配信を寄せる」損得分析（2026-06-03）

**前提となる技術事実（コード確認済み）:**
- Harnessの配信は `apps/worker/src/services/broadcast.ts` 等で **LINE Messaging APIの multicast / push を使用**。
- → **multicast/push も公式アカウントの月間通数枠・従量課金を消費する。同じチャネルなので通数カウンターは1つ。**
- → **「Harnessに寄せれば通数上限から逃げられる」は成立しない。**

**メリット（Harnessでしかできない＝AI自動化の射程）:**
- API/プログラム制御 → Claude/cronが人間ゼロで配信実行（「誰がいなくなっても回る」の本丸）
- ポイント×配信の連動 → 「失効30日前の保有者だけに償還促し」等、文脈付き配信
- きめ細かいセグメント → metadata/購入履歴/ランクで動的抽出
- 名寄せパーソナライズ → LINE×Shopify×メールを1顧客として最適化

**デメリット（代償）:**
- 通数上限は回避不能（期待してはいけない）
- 配信ミスの被害が桁違い：API一発で2万人、取り消せない。AI誤爆が即全顧客へ 🔴
- データ品質に全依存：今日のisFollowingバグのようにDBが壊れると配信が壊れる 🔴
- 保守の属人化：Worker(🔴保護ゾーン)を理解する人が要る＝哲学に逆行
- LINE仕様変更に自前追従が必要

**結論（推奨）: 段階的に寄せる。**
1. 今〜：配信実行は**公式**。Harnessは**ポイント・CRM・名寄せの頭脳**に徹する
2. データ健全化後：isFollowingバグ修正でHarness DBを信頼できる状態に
3. 将来：「失効前ポイント償還」等**Harnessにしかできない文脈配信から**、低リスクで段階的にAPI自動化を解禁
- 原則：**AI自動化の価値は"全部自動送信"ではなく"人手では回らない精密配信の自動化"。** マス配信は公式＋人間、精密配信はHarness＋AI。

## 4.8 【調査確定】LINE公式アカウントは2つ + 友だち数3値の正体（2026-06-03）

**公式アカウントが2つ存在:**
| アカウント | basicId | channelId | 用途 | Harness友だち | 今月配信 |
|---|---|---|---|---|---|
| ORYZAE《オリゼ》 | @055dpflo | 1655251148 | メインD2C | 6,829 | **0通** |
| フードコスメORYZAE 楽天市場店 | @603dsbmv | 2004907805 | 楽天店 | 12 | **0通** |

**友だち数3値の正体（完全決着）:**
| 数字 | 出どころ | 意味 |
|---|---|---|
| 20,134 | Harness count（param無し） | 全アカ・ブロック込みの累計総数（無意味） |
| 6,829 | Harness stats / `?lineAccountId=`指定 | メインのHarness把握分 |
| **15,126** | **公式管理画面** | **メインの真の友だち数（これが正）** |

→ **Harnessはメインの友だちを6,829人しか持っていない。公式は15,126人。実数の半分以下。**
→ 2026-04-09インポートは isFollowingバグだけでなく**取り込み件数自体が不足**。Harnessデータは想定以上に壊れている。

**【Phase 0の答え】「配信0件」の真因 = 通数上限ではなく「今月一度も配信していない」だけ:**
- `messagesThisMonth: 0`（両アカウント）
- スクショの「通数上限で停止」警告は**過去の履歴**であって今月の実績ではない。
- ⚠️ ただし messagesThisMonth もHarness集計＝公式実数とズレる可能性。**公式管理画面の通数表示が唯一の真実**。要・小泉さん目視。

**設計への影響:**
- 配信再開（Phase 1）の障壁は「上限」ではなく「運用が動いていない」。箱はある。動かせば飛ぶ。
- ただし**Harnessでセグメント抽出すると6,829人にしか届かない**（公式GUIなら15,126人全員）。
  → Phase 1は公式GUIで全員配信が正しい。Harness経由配信はデータ修復まで封印。
